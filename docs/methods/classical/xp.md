# Extreme Programming (XP)

## Kernidee

Engineering-getriebene agile Methode: technische Disziplinen
(TDD, Pair Programming, CI, Refactoring, einfaches Design) sind
*konstitutiv* für Agilität — nicht optional. Software-Qualität wird
auf Stundenebene aufrechterhalten, nicht am Ende eines Sprints.

## Herkunft & Vordenker

Kent Beck (Chrysler C3-Projekt, 1996–1999). Buch *Extreme Programming Explained* (1999).
Co-Autoren: Ward Cunningham, Ron Jeffries.

## Zentrale Artefakte / Rituale / Rollen

- **Werte:** Communication, Simplicity, Feedback, Courage, Respect
- **Praktiken (Auszug):** Pair Programming, TDD, Continuous Integration,
  Collective Code Ownership, Refactoring, Simple Design, Small Releases,
  Whole Team, On-site Customer, Sustainable Pace, Coding Standards
- **Rollen:** Customer, Programmer, Coach, Tracker

## Kadenz

Wochen-Iterationen historisch, heute meist Continuous Delivery.
Releases klein und häufig.

## Stärken

- Hohe technische Qualität, niedrige Defect-Rate
- Schnelles Feedback durch CI, TDD, kurze Releases
- Nachhaltige Geschwindigkeit über Jahre (kein Burn-down)
- Engineering-Praktiken sind universell anwendbar — auch außerhalb XP

## Schwächen / Anti-Patterns

- Pair Programming organisatorisch schwer durchzusetzen
- On-site Customer in den meisten Enterprise-Settings unrealistisch
- "XP-Light" (nur einige Praktiken) verliert das Werte-Fundament
- Discovery-Schicht fehlt komplett

## Enterprise-Eignung (500+)

XP als Gesamtsystem selten adoptiert. **Einzelpraktiken** (TDD, CI/CD,
Refactoring, Pairing, Trunk-Based Development) sind aber Standard in
jeder reifen Engineering-Org. Heute meist über DORA/SPACE-Metriken gemessen.

## Verhältnis zum Outcome-Loop

Bildet das **Engineering-Fundament** unter jedem Delivery-Track.
Ohne XP-typische Praktiken brechen alle anderen Methoden zusammen,
sobald Code-Komplexität wächst.

## Quellen

- Beck & Andres: *Extreme Programming Explained* (2nd ed., 2004)
- Forsgren, Humble, Kim: *Accelerate* (Engineering-Praktiken als Outcome-Treiber)
